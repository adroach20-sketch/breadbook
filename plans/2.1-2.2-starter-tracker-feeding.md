# Features 2.1 and 2.2 -- Starter Tracker + Feeding Plan

## Multi-Perspective Debate

### Round 1: Initial Positions

**Head Chef (Sourdough Expert)**

The data model is everything. A starter activity level is not a simple state -- it is a curve. After feeding, activity follows a predictable pattern: dormant (0-2hrs) -> waking up (2-4hrs) -> active (4-6hrs) -> peak (6-8hrs) -> past peak (8+hrs). But these timings shift dramatically based on:
- Ambient temperature (a 68F kitchen vs 78F kitchen can double the timeline)
- Flour type (whole grain ferments faster than white)
- Feeding ratio (1:1:1 peaks faster than 1:5:5)
- Starter maturity (established starters are more predictable)

My position: We should track temperature with every feeding and use it to estimate activity. The estimated activity level on the dashboard card should not be a dumb timer -- it should factor in the most recent temperature. Formula: base peak time = 6hrs at 75F, adjust +/-1hr per 5F deviation.

For the feeding log, peak rise % and peak rise time are the critical fields. These two values, tracked over time, tell you everything about starter health. The sparkline should show peak rise % over the last 7 feedings, not 7 days.

For bake-driven feeding plans (2.2): if you want peak activity at mix time T, and your starter typically peaks at P hours after feeding, you need to feed at T-P. If the starter is dormant (not fed for 24+ hrs), add 1-2 wake-up feeds before the bake feed.

**UX Designer (Mobile-First)**

The Head Chef is right about the science, but wrong about the UX priority. Most sourdough bakers are not scientists. They want to glance at their phone and know: Do I need to feed my starter right now?

The dashboard card hierarchy should be:
1. Starter name (emotional connection -- people name their starters)
2. Health status -- big, color-coded: green/yellow/red. Not a percentage. Not a curve.
3. Next action -- Feed now / Fed 3hrs ago check in 5hrs / Ready to bake!
4. Quick-log button -- one tap to log feeding without leaving dashboard

The sparkline is nice-to-have but secondary. On mobile, a tiny line chart in a card is barely readable. Better to show a simple streak indicator.

For the feeding form: temperature and peak rise % are useful but should be optional. The minimum viable log is: I fed my starter. Tap. Done. Everything else is friction for the 80% case.

Notifications are the real product here. A feeding reminder that arrives at the right time is worth more than the entire activity chart. But we need to be careful:
- Browser notifications are unreliable (user has to have the tab open or PWA installed)
- Push via service worker is better but we do not have PWA yet
- Start with browser notifications + in-app overdue badge
- Quiet hours are essential

Chef to UX: If you make the log too simple, the activity estimation becomes garbage.
UX to Chef: And if you make the log too complex, nobody logs anything. Zero data is worse than imprecise data.

---

### Round 2: Finding Consensus

**Activity Estimation -- Agreed Approach:**
Use a time-based estimation with optional temperature refinement. Default assumption: peak at 6 hours post-feed at ~72F room temp. If the user logs temperature, adjust the curve.

Activity states: Dormant (>12hrs), Waking Up (0-2hrs), Active (2-5hrs), Peak (5-8hrs), Past Peak (8-12hrs).

**Quick-Log UX -- Agreed Approach:**
Two-tier logging: quick log (1 tap, timestamp only) and full log (expandable form with water, flour, temperature, peak rise, notes, photo).

**Chart Library -- Agreed: Recharts** (new dependency, added to package.json)

**Notification Strategy -- Agreed:** Browser Notification API + in-app overdue badge. Quiet hours in Zustand store.

**Calendar View -- Agreed:** Simple 7-day horizontal strip. Each day shows feeding dots.

**Bake-Driven Feeding Plan:** User selects recipe + target bake time. System calculates feed times working backward from peak.

### Round 3: Final Architecture

See 005_starter_tracker.sql for database schema (starters, starter_logs, starter_schedules).

Component breakdown lives in src/features/starter/. Hooks in src/hooks/. Store in src/store/starterNotifications.ts. Types added to src/data/types.ts.

New dependency: recharts

Key Design Decisions:
- Activity estimation uses time-since-feed with temperature adjustment when available
- Quick-log is 1 tap, full log is expandable
- Sparkline shows last 7 feedings peak rise %
- 7-day horizontal calendar, not full month
- Browser notifications only (no service worker yet)
- Quiet hours in Zustand store
- All Supabase queries follow existing patterns (direct client calls, RLS-protected)
- Feature lives in src/features/starter/ following existing feature directory convention
