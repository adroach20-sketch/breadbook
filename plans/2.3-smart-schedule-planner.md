# Feature 2.3 — Smart Schedule Planner

## Multi-Round Debate

---

### Round 1: Head Chef vs UX Designer

**Head Chef (Fermentation Science)**

The scheduling algorithm is the heart of this feature, and getting it wrong means burnt trust. Here is what matters:

1. **Starter readiness is the biggest variable.** A dormant starter needs 2-3 feeds over 24-48 hours before it is bake-ready. A recently-fed starter might need one refresh. An active starter at peak is ready now. Each feed cycle is ~4-12 hours depending on temperature — this is NOT a fixed number.

2. **Temperature coefficients for bulk ferment are well-documented but approximate:**
   - 65F (18C): ~6-8 hours bulk
   - 70F (21C): ~4-6 hours bulk (baseline)
   - 75F (24C): ~3-4 hours bulk
   - 80F (27C): ~2-3 hours bulk
   - The relationship is roughly: every 5F above 70 cuts bulk time by ~25%. Below 70, add ~25% per 5F.

3. **Cold proof vs room temp proof is a binary fork in the timeline.** Cold proof = 1-48 hours in fridge (flexible). Room temp proof = 1-2 hours (time-sensitive). No-fridge bakes compress the entire timeline.

4. **Starter feed timing matters for the schedule.** Starter should be at peak (doubled, bubbly) when we mix. Peak is typically 4-6 hours after feeding at room temp, 8-12 hours in a cooler kitchen.

5. **Buffer times are critical.** Shaping takes 10-20 minutes. Scoring takes 5 minutes. Preheat is 30-60 minutes. Cool time is 1 hour minimum. These "small" steps add up to 2+ hours.

**Concerns:** Do not over-engineer the temperature math. Sourdough is not exact science — presenting times with false precision misleads beginners. Better to round to 15-minute increments.

---

**UX Designer (Mobile Flow & Visualization)**

1. **The "Plan a Bake" flow must be dead simple — one scrollable form, not a multi-step wizard.**
   - Section 1: Pick recipe + target eat time
   - Section 2: Starter status + kitchen conditions
   - Button: "Generate Schedule"
   - Result: Timeline view on same page or new view

2. **Timeline visualization on mobile.** A vertical timeline with color-coded steps works best:
   - Clock time (e.g., "Thu 8:00 AM")
   - Step name + duration
   - Color dot for category (green = starter prep, blue = dough work, orange = proofing, red = baking)
   - Expandable detail

3. **One-tap actions on each step:** "Set Reminder" — keep secondary.

4. **"Add to Bake Mode" is the golden CTA.** One tap loads the recipe into Guided Bake Mode.

5. **Warnings need to be warm, not scary.** "This recipe needs about 12 hours. Want to adjust your eat time, or pick a same-day recipe?"

**Concerns:** Starter status input is fragile. Most beginners don't know "active" vs "recently fed." Use plain language: "When did you last feed your starter?"

### Round 2: Addressing Critiques & Algorithm Design

**Head Chef responds to UX concerns:**

Agreed on starter status language. Better model:

| User sees | Internal status | Extra feeds needed | Feed time added |
|---|---|---|---|
| "It is bubbly and doubled right now!" | peak | 0 | 0 hours |
| "I fed it a few hours ago" | recently_fed | 0 | 2-4 hours (wait for peak) |
| "I fed it yesterday, it has fallen back" | fed_but_fallen | 1 | 6-8 hours |
| "It has been in the fridge for a while" | dormant | 2 | 24 hours (two feeds) |

Temperature coefficient — simple linear model:

```
adjustmentFactor = 1 + (70 - roomTempF) * 0.05
adjustedMinutes = baseMinutes * adjustmentFactor
```

At 75F: factor = 0.75 (25% shorter). At 65F: factor = 1.25. At 80F: factor = 0.50.
Round to nearest 15 minutes.

**UX Designer responds to Chef concerns:**

Timeline display strategy:
- **Active steps** (mix, fold, shape, score): Show exact scheduled time + "~15 min"
- **Passive steps** (bulk ferment, cold proof, cooling): Show start time + duration
- **Time-critical steps** (bake): Show exact time + exact duration

Warning check: if total timeline is < 80% of recipe base time, or any step ends after the next starts, show a yellow warning.

**Joint decision on algorithm flow:**

1. Start with target eat time
2. Subtract cool time
3. Subtract bake time(s)
4. Subtract preheat time
5. Subtract proof time (cold if fridge, else room temp 90 min)
6. Subtract shape time
7. Subtract bulk ferment time (temp-adjusted)
8. Subtract stretch & fold / active work time
9. Subtract autolyse / mix time
10. Subtract levain build time (if applicable)
11. Subtract starter prep time (based on status)
12. Earliest timestamp = "Start here"

### Round 3: Final Architecture & Implementation Plan

**Consensus**

New feature at `/schedule` with sub-views:
1. **Schedule Form** (`/schedule/new`) — single-page form
2. **Schedule View** (`/schedule/:id` or inline after generation) — vertical timeline
3. **Schedule History** (`/schedule`) — list of saved schedules

**Component Breakdown:**

```
src/features/schedule/
  ScheduleForm.tsx        — Form: recipe picker, datetime, starter, conditions
  ScheduleTimeline.tsx    — Generated vertical timeline with color-coded steps
  ScheduleStepCard.tsx    — Individual step card in timeline
  ScheduleHistory.tsx     — List of saved schedules
src/lib/
  schedule-engine.ts      — Pure functions: reverse-timeline calculation, temp adjustments
src/data/types.ts         — New types added
supabase/migrations/
  006_bake_schedules.sql  — Database table
```

**Database: bake_schedules**

- id uuid PK
- user_id uuid FK -> auth.users
- recipe_id text NOT NULL (supports uuid and bb-* local IDs)
- recipe_title text NOT NULL (denormalized)
- target_eat_time timestamptz
- starter_name text
- starter_status text
- room_temp_f integer
- fridge_available boolean DEFAULT true
- schedule_steps jsonb NOT NULL
- warnings jsonb DEFAULT '[]'
- created_at, updated_at timestamptz

**Starter Tracker Integration:**

Manual mode (default): text input for name + status picker buttons.
Linked mode (future): when starters table exists, pull from it.
Check for starters table at form load — graceful fallback.

**Implementation Order:**

1. Types (schedule types in types.ts)
2. Migration (006_bake_schedules.sql)
3. Schedule engine (pure logic, testable)
4. Schedule Form page
5. Schedule Timeline component
6. Schedule History page
7. Routing + nav updates

**Scope control — NOT building:**
- Set Reminder (requires push notification infra)
- Screenshot sharing (requires html2canvas dependency)
- Link sharing (requires public URLs)
- Pre-populating times into Bake Mode UI (that is feature 1.13)
