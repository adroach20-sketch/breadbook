# Community Features Plan: 3.1 Recipe Sharing + 3.2 Bake Feed + 3.3 Baker Profiles

## Multi-Round Debate

### Round 1: Initial Perspectives

**Head Chef (Data & Backend)**

Key concerns:

1. **Data model for social features.** We need new tables: `recipe_likes`, `follows`, `feed_posts`, `feed_comments`, plus expanding `profiles` and `bake_logs`. The `recipe_saves` table is being created by another agent (migration 004) -- we reference it, not recreate it.

2. **RLS policies are the heart of safety.** Public profiles need SELECT for everyone but UPDATE only for owner. Feed posts need the same. Comments visible to all, insertable by authenticated users, deletable by comment owner OR post owner. Follows: authenticated users can insert/delete their own rows.

3. **Privacy defaults must be conservative.** Profiles should default to `is_public = false`. Recipes default to `is_public = false` (already the case). Bake logs should NOT auto-share -- a user must explicitly create a "feed post" from a bake log. No one's baking failures should leak into a public feed without consent.

4. **Content moderation.** For MVP, we don't need AI moderation. Keep it simple: report button on posts/comments (stores in a `reports` table for manual review). But honestly, even that might be over-scoped for v0.2. The sourdough community is inherently wholesome. Defer reports table -- just build the sharing features.

5. **Feed algorithm.** Do NOT build an algorithmic feed. Chronological, reverse-chronological. Period. This is a sourdough app, not Twitter. The "algorithm" is: show posts from people you follow, newest first. With a separate "discover" tab showing all public posts.

6. **Username uniqueness and URL structure.** `@username` profiles need unique, URL-safe usernames. The `profiles` table already has `username text unique`. We need to add validation: lowercase, alphanumeric + underscores, 3-30 chars.

**UX Designer (Experience & Mobile)**

Key concerns:

1. **Community warmth.** This is the defining UX challenge. Social features in a sourdough app should feel like walking into a warm bakery, not scrolling Instagram. The feed should celebrate process, not perfection. Design choices: warm color palette (already have it), generous whitespace, no follower counts on the main feed, photo-first cards.

2. **Mobile feed scrolling.** The feed must be butter-smooth on mobile. Single-column layout, photo cards with generous tap targets. No infinite scroll -- paginate with "Load more" button (simpler, more intentional). Skeleton loaders while loading.

3. **Profile discoverability.** How do people find other bakers? Three paths: (a) see their name on a recipe card in the public feed, (b) see their name on a bake post, (c) search (defer for now). Profile pages show: avatar, bio, starter name, bake count, public recipes, recent bakes. Simple.

4. **Social proof without toxicity.** Show like counts on recipes but keep them subtle (small heart + number). Don't show follower counts prominently on profiles -- show "bakes this month" instead. The status signal is "I bake a lot" not "I have followers." No public like lists.

5. **Share flow.** Making a recipe public should be a simple toggle, not a multi-step process. Sharing a bake to the feed should be one tap from the journal detail page: "Share to Feed" button that creates a feed post with the bake's photo and notes.

6. **Feed empty states matter.** When a new user first visits the feed, it shouldn't feel hopeless. Show encouraging copy with a CTA to browse recipes or share their first bake.

---

### Round 2: Critiques & Consensus

**Head Chef responds to UX:**

- Agree on no follower counts in the feed. The data model will track follows for the "following" feed filter, but we won't expose counts prominently.
- "Share to Feed" from journal detail is clean. The `feed_posts` table links to `bake_logs` via `bake_log_id`. Post content = bake log photo + optional caption.
- Empty state: "Be the first to share a bake! Your community is just getting started." with a CTA to journal. No fake seed content.
- Username validation: enforce on profile edit. Add a check constraint in migration.

**UX Designer responds to Chef:**

- Accept the empty state approach -- authenticity matters for this brand. But add a "Discover recipes" link in the empty feed state so there's always somewhere to go.
- Privacy defaults of `is_public = false` are correct. But the toggle to make a profile public should be prominent and encouraging, not buried in settings. Put it right on the profile edit page with copy like "Let other bakers find you."
- Agree on chronological feed. But we should have two tabs: "Following" (posts from people you follow) and "Discover" (all public posts, newest first). If you follow nobody, the Following tab shows a friendly nudge.

**Consensus on key decisions:**

1. **Feed algorithm:** Chronological, two tabs (Following / Discover)
2. **Privacy defaults:** Everything private by default, easy toggles to go public
3. **Moderation:** Defer. Sourdough community is low-risk for MVP.
4. **Profile URL:** `/@username` route, username validated (lowercase, alphanum + underscore, 3-30 chars)
5. **Follower counts:** Track but don't display prominently. Show "bakes this month" as the primary social proof.
6. **Empty states:** Warm, encouraging copy with CTAs -- no fake content.
